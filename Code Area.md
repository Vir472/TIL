# (Code/Text Segment)

## 핵심 개념

- 프로세스 메모리의 4가지 영역 중 하나
- 프로그램이 실행할 **기계어 코드(명령어)**가 저장되는 공간
- CPU가 이 영역의 명령어를 순서대로 읽어서 실행함

## 특징

- **읽기 전용(Read-Only)**: 실행 중에 코드가 변경되면 안 되므로 쓰기 불가
- **공유 가능**: 같은 프로그램을 여러 프로세스가 실행할 때 코드 영역은 공유 가능 (메모리 절약)
- 프로그램 시작 시 로드되고 종료 시 해제됨

## 코드가 기계어로 변환되는 과정

```
C# 소스코드 (.cs)
    ↓ 컴파일
IL 코드 (중간 언어, .dll)
    ↓ JIT 컴파일 (실행 시점)
기계어 → Code 영역에 적재
    ↓
CPU가 실행
```

## C#이 특이한 이유

- C#은 바로 기계어로 컴파일되지 않고 **IL(Intermediate Language)** 로 먼저 변환됨
- 실행 시 CLR(Common Language Runtime)의 JIT 컴파일러가 기계어로 변환
- Unity는 여기에 추가로 IL2CPP 옵션으로 아예 C++ 코드로 변환 후 빌드 가능

## 게임 개발 관점에서

- **Mono vs IL2CPP**: Unity 빌드 시 두 가지 스크립팅 백엔드 선택 가능
  - `Mono`: JIT 방식. 빌드 빠름, 런타임에 기계어 변환 → 초기 실행 느릴 수 있음
  - `IL2CPP`: IL을 C++로 변환 후 AOT(Ahead-Of-Time) 컴파일 → 실행 성능 향상, 역공학 방지
- **모바일/콘솔 배포**: iOS는 JIT 실행 자체가 금지되어 있어 IL2CPP 필수
- **핫 리로딩**: Unity 에디터에서 코드 수정 후 바로 반영되는 게 가능한 이유 — Mono의 JIT 특성 덕분
- **코드 보안**: IL2CPP를 쓰면 코드 영역이 기계어로 변환되어 있어 IL 디컴파일러로 소스코드 복원이 어려워짐 → 게임 핵/치트 방지에 유리

## 의문점 / 더 알아볼 것
