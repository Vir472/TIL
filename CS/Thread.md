# 스레드 (Thread)

## 핵심 개념

- **스레드(Thread)**: 프로세스 안에서 실제로 실행되는 작업의 단위
- 프로세스가 "공장"이라면 스레드는 "공장 안의 일꾼"
- 같은 프로세스 안의 스레드들은 **Code, Data, Heap을 공유**하고 **Stack만 각자 독립적**으로 가짐

## 스레드가 가지는 것

```
프로세스 (공유 영역)
├─ Code 영역  ┐
├─ Data 영역  ├─ 모든 스레드가 공유
├─ Heap 영역  ┘
│
├─ 스레드 1 ── 독립적인 Stack + 레지스터 + PC(Program Counter)
├─ 스레드 2 ── 독립적인 Stack + 레지스터 + PC
└─ 스레드 3 ── 독립적인 Stack + 레지스터 + PC
```

- Stack이 독립적인 이유: 각 스레드는 자기만의 함수 호출 흐름을 가짐
- PC(Program Counter): "지금 어디 실행 중인지" 위치를 기억하는 레지스터

## 멀티스레드의 장단점

| 항목 | 내용                                                |
| ---- | --------------------------------------------------- |
| 장점 | CPU 코어를 병렬로 활용해 성능 향상                  |
| 장점 | 한 스레드가 I/O 대기 중에도 다른 스레드가 실행 가능 |
| 장점 | 프로세스보다 생성/전환 비용이 작음                  |
| 단점 | 공유 메모리 접근 시 동기화 문제 발생 가능           |
| 단점 | 디버깅이 어려움 (실행 순서가 비결정적)              |
| 단점 | 하나의 스레드 크래시가 전체 프로세스에 영향         |

## 멀티스레드에서 발생하는 문제들

공유 메모리를 여러 스레드가 동시에 접근할 때 생기는 문제들

- **레이스 컨디션(Race Condition)**: 두 스레드가 같은 데이터에 동시에 읽기/쓰기를 해서 결과가 실행 순서에 따라 달라지는 문제
- **데드락(Deadlock)**: 두 스레드가 서로 상대방의 잠금이 풀리길 기다리며 영원히 멈추는 상태
- **기아(Starvation)**: 특정 스레드가 계속 CPU 시간을 할당받지 못하고 대기만 하는 상태
- **라이브락(Livelock)**: 데드락과 달리 스레드가 계속 실행되지만 서로 양보만 반복해서 진행이 안 되는 상태

## 게임 개발 관점에서

- **Unity 메인 스레드 제약**: Unity API(GameObject, Transform 등)는 메인 스레드에서만 호출 가능. 백그라운드 스레드에서 호출 시 크래시
- **Unity 스레드 구조**:
  | 스레드 | 역할 |
  |--------|------|
  | 메인 스레드 | 게임 로직, Update() 실행, Unity API 호출 |
  | 렌더 스레드 | GPU에 렌더링 명령 전달 |
  | 워커 스레드 | Job System으로 분배된 연산 처리 |
  | 오디오 스레드 | 사운드 믹싱 및 재생 |

- **Unity Job System**: Unity가 제공하는 안전한 멀티스레드 솔루션. struct 기반 데이터만 다루도록 강제해서 레이스 컨디션을 구조적으로 방지
