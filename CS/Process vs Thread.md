# 프로세스 vs 스레드

## 한 줄 정리

- **프로세스**: 실행 중인 프로그램 자체. 독립적인 메모리 공간을 가짐
- **스레드**: 프로세스 안에서 실제로 실행되는 작업 단위. 메모리를 공유함

## 메모리 구조 비교

```
프로세스 A                      프로세스 B
┌─────────────────┐             ┌─────────────────┐
│  Code 영역       │             │  Code 영역       │
│  Data 영역       │             │  Data 영역       │
│  Heap 영역       │             │  Heap 영역       │
│                 │             │                 │
│  ┌───────────┐  │             │  ┌───────────┐  │
│  │ 스레드 1    │  │             │  │ 스레드 1   │  │
│  │ (Stack)   │  │             │  │ (Stack)   │  │
│  ├───────────┤  │             │  ├───────────┤  │
│  │ 스레드 2    │  │             │  │ 스레드 2   │  │
│  │ (Stack)   │  │             │  │ (Stack)   │  │
│  └───────────┘  │             │  └───────────┘  │
└─────────────────┘             └─────────────────┘
        ↑                               ↑
   메모리 완전 분리               메모리 완전 분리
   (프로세스 간 공유 불가)
```

프로세스 안의 스레드들은 Code, Data, Heap을 공유하고 Stack만 각자 독립적으로 가짐

## 핵심 차이 비교

| 항목            | 프로세스                              | 스레드                              |
| --------------- | ------------------------------------- | ----------------------------------- |
| 정의            | 실행 중인 프로그램 인스턴스           | 프로세스 안의 실행 단위             |
| 메모리          | 독립적인 공간 (완전 분리)             | 프로세스 메모리 공유 (Stack만 독립) |
| 생성 비용       | 무거움                                | 가벼움                              |
| 컨텍스트 스위칭 | 비용 큼 (메모리 공간 전환)            | 비용 작음 (같은 메모리 공간)        |
| 통신 방식       | IPC (파이프, 소켓, 공유 메모리 등)    | 공유 메모리 직접 접근               |
| 독립성          | 하나가 죽어도 다른 프로세스 영향 없음 | 하나가 죽으면 프로세스 전체 영향    |
| 동기화 필요     | 상대적으로 덜 필요                    | 공유 자원 접근 시 반드시 필요       |
| 보안            | 메모리 분리로 안전                    | 공유 메모리로 취약할 수 있음        |

## 언제 프로세스를 쓰고 언제 스레드를 쓰냐면

**프로세스가 적합한 경우**

- 완전한 격리가 필요할 때 (하나가 죽어도 다른 것에 영향 없어야 할 때)
- 게임 서버처럼 크래시 격리가 중요한 경우
- 서로 다른 프로그램이 협력하는 경우 (게임 클라이언트 + 런처)

**스레드가 적합한 경우**

- 같은 데이터를 공유하며 병렬로 처리해야 할 때
- 생성/전환 비용을 줄여야 할 때
- UI가 멈추지 않게 백그라운드 작업이 필요할 때 (게임 로딩, 시뮬레이션 등)

## 비유로 정리

```
프로세스 = 독립된 주방
  → 각자 냉장고, 조리도구, 재료가 따로 있음
  → 한 주방에 불이 나도 옆 주방은 멀쩡

스레드 = 같은 주방의 요리사들
  → 냉장고, 조리도구를 함께 씀
  → 한 명이 재료를 잘못 건드리면 다른 사람 요리에도 영향
  → 하지만 소통이 빠르고 효율적
```

## 게임 개발 관점에서

- **게임 자체는 하나의 프로세스**: 그 안에서 렌더링·물리·오디오·게임로직이 각각 스레드로 나뉨
- **런처 + 게임 클라이언트**: 별도 프로세스로 분리. 런처가 죽어도 게임은 살아있음
- **Unity 구조**:

```
  Unity 프로세스
  ├─ 메인 스레드   — 게임 로직, Update()
  ├─ 렌더 스레드  — GPU 커맨드 전송
  ├─ 오디오 스레드 — 사운드 처리
  └─ 워커 스레드  — Job System 작업
```

## 의문점 / 더 알아볼 것

- 멀티프로세스 vs 멀티스레드 — 게임 서버 설계 시 어떤 기준으로 선택하는지
- 컨텍스트 스위칭이 실제 게임 성능에 미치는 영향 측정 방법
