# CPU 레지스터 (Register)

## 관련 노트

→ [[PCB]] — 컨텍스트 스위칭 시 레지스터 값을 저장하는 자료구조
→ [[프로세스 메모리 영역]] — 메모리 계층과의 관계

## 핵심 개념

- CPU 내부에 있는 **가장 빠른 저장 공간**
- RAM보다 수십~수백 배 빠름 — CPU가 연산할 때 직접 접근하는 임시 저장소
- 용량이 매우 작음 (수십 바이트 ~ 수 KB 수준)

## 메모리 계층에서의 위치

```
빠름 / 비쌈 / 작음
┌─────────────────┐
│    레지스터       │  ← CPU 내부, 1 사이클 접근
├─────────────────┤
│   L1 캐시        │  ← 수 사이클
├─────────────────┤
│   L2 캐시        │
├─────────────────┤
│   L3 캐시        │
├─────────────────┤
│      RAM        │  ← 수백 사이클
├─────────────────┤
│      SSD        │
└─────────────────┘
느림 / 쌈 / 큼
```

## 주요 레지스터 종류

| 레지스터        | 약자 | 역할                                      |
| --------------- | ---- | ----------------------------------------- |
| Program Counter | PC   | 다음 실행할 명령어의 메모리 주소          |
| Stack Pointer   | SP   | 현재 Stack의 최상단 주소                  |
| Base Pointer    | BP   | 현재 함수의 Stack Frame 시작 주소         |
| Accumulator     | ACC  | 산술/논리 연산 결과를 임시 저장           |
| Index Register  | IR   | 배열 인덱싱, 반복문 카운터                |
| Flags Register  | FR   | 연산 결과 상태 (Zero, Overflow, Carry 등) |

## 컨텍스트 스위칭과의 관계

레지스터가 중요한 이유 — 컨텍스트 스위칭 시 저장/복원 대상

```
프로세스 A 실행 중
  PC = 0x00A1F2  (여기까지 실행함)
  SP = 0x7FFF80  (스택 위치)
  ACC = 42       (계산 중이던 값)
    ↓ 컨텍스트 스위칭
  → A의 레지스터 값 전부 PCB에 저장
  → B의 PCB에서 레지스터 값 복원
프로세스 B 실행 재개
```

레지스터 값이 저장/복원되기 때문에 프로세스가 중단됐다가 재개해도 이어서 실행 가능

## 게임 개발 관점에서

- **직접 다룰 일 없음**: 고수준 언어(C#, C++)에서는 컴파일러가 레지스터를 자동으로 관리
- **PC(Program Counter)가 중요한 이유**: 각 스레드가 독립적인 PC를 갖기 때문에 스레드마다 "지금 어디 실행 중인지"를 따로 추적 가능 → 병렬 실행의 근거
- **SIMD 레지스터**: 현대 CPU는 XMM, YMM 같은 넓은 레지스터로 여러 데이터를 한 번에 처리. 게임 물리 연산, 벡터 수학 최적화에 활용
- **Unity Burst Compiler**: C# 코드를 SIMD 레지스터를 활용하는 네이티브 코드로 변환해 Job System 성능을 극대화하는 원리

## 의문점 / 더 알아볼 것

- x86-64 아키텍처의 범용 레지스터 (RAX, RBX, RCX, RDX 등) 명명 규칙
- SIMD(Single Instruction Multiple Data)가 게임 물리 연산을 빠르게 만드는 원리
- ARM 아키텍처(모바일/Apple Silicon)의 레지스터 구조가 x86과 다른 점
