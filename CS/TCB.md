# TCB (Thread Control Block)

## 핵심 개념

- **스레드 제어 블록(Thread Control Block)**: OS가 스레드를 관리하기 위해 유지하는 자료구조
- PCB가 프로세스 단위 관리 구조라면, TCB는 스레드 단위 관리 구조
- 스레드가 생성될 때 OS가 만들고, 스레드가 종료될 때 삭제됨
- PCB 안에 포함되거나 PCB와 연결된 형태로 존재함

## TCB에 저장되는 정보

```
TCB
├─ 스레드 식별 정보
│   ├─ TID (Thread ID)        — 스레드 고유 번호
│   └─ 소속 PID               — 어느 프로세스에 속하는지
│
├─ CPU 레지스터 정보           — 컨텍스트 스위칭 시 저장/복원
│   ├─ PC (Program Counter)   — 다음 실행할 명령어 주소
│   ├─ SP (Stack Pointer)     — 현재 Stack 위치
│   └─ 범용 레지스터들
│
├─ 스레드 상태
│   └─ Running / Ready / Waiting / Terminated
│
├─ 스택 정보
│   └─ 스레드 전용 Stack 영역 포인터
│
└─ 스케줄링 정보
    ├─ 우선순위 (Priority)
    └─ 스케줄링 큐 포인터
```

## PCB와의 관계

```
PCB (프로세스 전체 관리)
├─ PID, 메모리 정보, 열린 파일 목록 ...
└─ 스레드 목록
    ├─ TCB 1 (스레드 1의 실행 상태)
    ├─ TCB 2 (스레드 2의 실행 상태)
    └─ TCB 3 (스레드 3의 실행 상태)
```

- PCB는 프로세스의 공유 자원(메모리, 파일 등) 전체를 관리
- TCB는 각 스레드의 독립적인 실행 상태(레지스터, Stack, PC)만 관리
- 같은 프로세스의 TCB들은 PCB를 통해 메모리·파일 등을 공유

## 스레드 컨텍스트 스위칭

프로세스 간 스위칭보다 비용이 작은 이유가 TCB 구조에 있음

```
프로세스 간 스위칭
→ PCB 저장/복원 + 메모리 공간 전환 (페이지 테이블 교체) → 비용 큼

같은 프로세스 내 스레드 간 스위칭
→ TCB 저장/복원만 하면 됨 (메모리 공간은 그대로) → 비용 작음
```

## 게임 개발 관점에서

- **직접 다룰 일은 없음**: TCB도 PCB와 마찬가지로 OS 커널이 관리. 게임 개발자가 직접 접근 불가
- **스레드를 많이 만들면 안 되는 이유**: 스레드마다 TCB가 생성되고 스케줄링 오버헤드 증가. Unity Job System이 스레드를 매번 새로 만들지 않고 워커 스레드 풀을 재사용하는 이유
- **같은 프로세스 내 스레드 스위칭이 빠른 이유**: TCB만 교체하면 되고 메모리 공간은 그대로라 컨텍스트 스위칭 비용이 프로세스 간 전환보다 훨씬 작음 → 게임에서 멀티스레드를 선호하는 근거

## 의문점 / 더 알아볼 것

- Linux에서 TCB에 해당하는 구조체는 무엇인지 (PCB와 동일한 `task_struct`를 씀)
- 유저 레벨 스레드와 커널 레벨 스레드에서 TCB 관리 방식의 차이
