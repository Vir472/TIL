# PCB (Process Control Block)

## 핵심 개념

- **프로세스 제어 블록(Process Control Block)**: OS가 프로세스를 관리하기 위해 유지하는 자료구조
- 프로세스가 생성될 때 OS가 만들고, 프로세스가 종료될 때 삭제됨
- OS의 커널 메모리 영역에 저장됨 — 사용자 프로그램이 직접 접근 불가

## PCB에 저장되는 정보

```
PCB
├─ 프로세스 식별 정보
│   ├─ PID (Process ID)       — 프로세스 고유 번호
│   └─ PPID (Parent PID)      — 부모 프로세스 ID
│
├─ 프로세스 상태
│   └─ New / Ready / Running / Waiting / Terminated
│
├─ CPU 레지스터 정보           — 컨텍스트 스위칭 시 저장/복원
│   ├─ PC (Program Counter)   — 다음 실행할 명령어 주소
│   ├─ SP (Stack Pointer)     — 현재 Stack 위치
│   └─ 범용 레지스터들
│
├─ 스케줄링 정보
│   ├─ 우선순위 (Priority)
│   └─ 스케줄링 큐 포인터
│
├─ 메모리 관리 정보
│   ├─ 베이스/리밋 레지스터
│   └─ 페이지 테이블 포인터
│
├─ I/O 상태 정보
│   └─ 열린 파일 목록, 장치 목록
│
└─ 계정 정보
    ├─ CPU 사용 시간
    └─ 생성 시간
```

## PCB가 필요한 이유 — 컨텍스트 스위칭

CPU는 한 번에 하나의 프로세스만 실행할 수 있음. 여러 프로세스를 번갈아 실행하려면 "지금 하던 작업을 어딘가에 저장"해야 함 → PCB가 그 저장소

```
프로세스 A 실행 중
    ↓ 타이머 인터럽트 or I/O 대기
A의 현재 상태(레지스터, PC 등) → A의 PCB에 저장
    ↓
B의 PCB에서 저장된 상태 복원
    ↓
프로세스 B 실행 재개
    ↓ 다시 스위칭
B의 상태 → B의 PCB에 저장
A의 PCB 복원 → A 실행 재개
```

PCB가 없으면 컨텍스트 스위칭 자체가 불가능 → 멀티태스킹 불가

## 프로세스 상태와 PCB

```
생성(New)
  ↓ PCB 생성
준비(Ready) ──── 스케줄러 선택 ────→ 실행(Running)
  ↑                                      │
  │         I/O 완료                      │ I/O 요청 or
  └──────── 대기(Waiting) ←───────────────┘ 타임아웃
                                             │
                                           종료(Terminated)
                                             ↓
                                           PCB 삭제
```

## 게임 개발 관점에서

- **직접 다룰 일은 없음**: PCB는 OS 커널이 관리하는 구조라 게임 개발자가 직접 접근하거나 수정할 수 없음
- **간접적으로 영향받는 부분**:
  - 컨텍스트 스위칭 비용이 쌓이면 게임 성능 저하 → 스레드를 너무 많이 만들지 않아야 하는 이유
  - Unity의 Job System이 스레드를 직접 만들지 않고 워커 스레드 풀을 재사용하는 이유 — 스레드(TCB) 생성/삭제 비용 절감
  - `Process.GetCurrentProcess()`로 C#에서 현재 프로세스 정보(PID, 메모리 사용량 등) 조회 가능 — 게임 퍼포먼스 모니터링에 활용
- **게임 엔진 내부**: Unity 에디터 자체도 하나의 프로세스. 에디터와 게임 플레이어가 별도 프로세스로 분리되는 이유도 PCB 단위 격리 덕분

## 의문점 / 더 알아볼 것

- 컨텍스트 스위칭 시 PCB 저장/복원에 걸리는 실제 비용은 얼마나 되는지
- 스레드가 많아질수록 TCB도 늘어나고 스케줄링 오버헤드가 증가하는 구조
- Linux에서 PCB에 해당하는 `task_struct` 구조체
