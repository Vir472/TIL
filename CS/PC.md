# Program Counter (PC)

## 핵심 개념

- **Program Counter(PC)**: 다음에 실행할 명령어의 메모리 주소를 저장하는 레지스터
- **Instruction Pointer(IP)** 라고도 불림 (x86 아키텍처에서는 IP/EIP/RIP)
- CPU는 PC가 가리키는 주소의 명령어를 가져와 실행하고, PC를 자동으로 다음 명령어 주소로 증가시킴

## 동작 방식 — Fetch-Decode-Execute 사이클

CPU가 명령어를 실행하는 기본 사이클

```
① Fetch   — PC가 가리키는 주소에서 명령어를 가져옴
② Decode  — 명령어가 무엇인지 해석
③ Execute — 명령어 실행
④ PC 갱신 — PC를 다음 명령어 주소로 증가
    → ①로 돌아가 반복
```

```
메모리 주소    명령어
0x00A0  →  LOAD R1, 42     ← PC = 0x00A0, 이 명령어 실행
0x00A4  →  ADD  R1, R2     ← 실행 후 PC = 0x00A4로 증가
0x00A8  →  STORE R1, 0xFF  ← 실행 후 PC = 0x00A8로 증가
```

## PC가 단순히 증가만 하지 않는 경우

```
일반 실행     → PC += 명령어 크기 (순차 증가)
함수 호출     → PC = 함수 시작 주소로 점프
return        → PC = 호출 전 복귀 주소로 복원
if/else 분기  → 조건에 따라 PC를 다른 주소로 점프
반복문        → 조건 불만족 시 PC를 루프 시작 주소로 점프
인터럽트 발생  → PC = 인터럽트 핸들러 주소로 점프 + 기존 PC 저장
```

```csharp
// 이 C# 코드가 실행될 때 PC의 움직임
int x = 10;          // PC → 다음 줄 주소로 증가
if (x > 5)           // 조건 평가 → true면 아래로, false면 else로 PC 점프
{
    x = x * 2;       // PC → 다음 줄
}
// x = 20
```

## 스레드마다 PC가 독립적인 이유

```
프로세스 (메모리 공유)
├─ 스레드 1 — PC: 0x00A4  ← 함수 A의 3번째 줄 실행 중
├─ 스레드 2 — PC: 0x01F8  ← 함수 B의 7번째 줄 실행 중
└─ 스레드 3 — PC: 0x0320  ← 함수 C의 1번째 줄 실행 중
```

- 각 스레드가 독립적인 PC를 가지기 때문에 서로 다른 코드를 동시에 실행 가능
- PC가 공유되면 두 스레드가 같은 명령어 위치를 실행하게 되어 병렬 실행 불가능
- [[TCB]]에 스레드별 PC 값이 저장됨

## 컨텍스트 스위칭에서 PC의 역할

```
스레드 A 실행 중
  PC = 0x00A4  ← "여기까지 실행했음"
    ↓ 컨텍스트 스위칭 발생
  PC 값 0x00A4 → TCB에 저장
    ↓
  스레드 B의 TCB에서 PC 값 복원 → 0x01F8
  스레드 B가 0x01F8부터 실행 재개
    ↓ 다시 스위칭
  스레드 A의 TCB에서 PC 값 복원 → 0x00A4
  스레드 A가 0x00A4부터 이어서 실행
```

PC가 정확히 저장/복원되기 때문에 스레드가 중단됐다가 재개해도 이어서 실행 가능

## 예외/인터럽트 처리와 PC

```
정상 실행 중
  PC = 0x00A4
    ↓ 예외 발생 (예: 0으로 나누기)
  기존 PC(0x00A4) → 스택에 저장
  PC → 예외 핸들러 주소로 변경
  예외 핸들러 실행
    ↓ 핸들러 종료
  스택에서 원래 PC(0x00A4) 복원
  0x00A4부터 실행 재개 (또는 프로세스 종료)
```

## 아키텍처별 PC 이름

| 아키텍처       | PC 레지스터 이름                   |
| -------------- | ---------------------------------- |
| x86 (32bit)    | EIP (Extended Instruction Pointer) |
| x86-64 (64bit) | RIP (Register Instruction Pointer) |
| ARM (32bit)    | R15 (PC)                           |
| ARM64 (64bit)  | PC                                 |
| MIPS           | PC                                 |

## 게임 개발 관점에서

- **직접 다룰 일 없음**: C#, C++에서는 PC를 직접 조작하지 않음. 컴파일러가 분기/점프 명령어로 변환
- **디버거와 PC**: Visual Studio 디버거에서 중단점(Breakpoint)을 걸면 해당 주소에서 PC가 멈춤. "다음에 실행될 줄"을 노란 화살표로 보여주는 것이 바로 PC가 가리키는 위치
- **예외 스택 트레이스**: 게임 크래시 시 출력되는 스택 트레이스가 각 함수 호출 시점의 PC 값(복귀 주소)을 역추적한 결과
- **Unity IL2CPP와 PC**: IL2CPP로 빌드하면 C# 코드가 C++를 거쳐 네이티브 기계어로 변환됨. 릴리즈 빌드에서 크래시 시 PC 주소만 알 수 있고 심볼 파일이 없으면 어느 코드인지 파악 불가 → 심볼 파일 보관이 중요한 이유
- **무한루프와 PC**: 게임에서 무한루프가 발생하면 PC가 같은 주소 범위를 계속 순환 → CPU 점유율 100% → 게임 프리징

## 의문점 / 더 알아볼 것

- 분기 예측(Branch Prediction) — CPU가 PC의 다음 값을 미리 예측해 성능을 올리는 기법
- 파이프라인(Pipeline) — Fetch/Decode/Execute를 겹쳐서 실행해 PC를 더 빠르게 처리하는 구조
- GOT(Global Offset Table), PLT — 동적 라이브러리에서 PC 기반 주소 계산
