# 프로세스 메모리 영역

## 핵심 개념

프로세스가 실행되면 OS가 메모리를 다음 4가지 영역으로 나눠서 할당함

```
높은 주소
┌─────────────┐
│    Stack    │  ← 지역변수, 함수 호출 정보 (자동 관리)
├─────────────┤
│      ↓      │  (Stack이 아래로 자람)
│             │
│      ↑      │  (Heap이 위로 자람)
├─────────────┤
│    Heap     │  ← 동적 할당 메모리 (수동/GC 관리)
├─────────────┤
│    Data     │  ← 전역변수, 정적변수
├─────────────┤
│    Code     │  ← 실행할 기계어 코드
└─────────────┘
낮은 주소
```

## 각 영역 요약

| 영역        | 저장 내용                | 관리 방식                   |
| ----------- | ------------------------ | --------------------------- |
| Code (Text) | 실행 코드 (기계어)       | OS가 관리, 읽기 전용        |
| Data        | 전역변수, static 변수    | 프로그램 시작~종료까지 유지 |
| Heap        | new로 동적 할당한 객체   | C#은 GC가 자동 관리         |
| Stack       | 지역변수, 함수 호출 스택 | 함수 호출/반환 시 자동 관리 |

## C# 관점에서

- `int x = 5;` (지역변수) → Stack
- `new GameObject()` → Heap (GC 대상)
- `static int count;` → Data 영역

## 게임 개발 관점에서

- **GC가 게임의 적인 이유**: Unity는 C# GC를 사용하는데, GC가 Heap을 정리하는 순간 게임이 일시적으로 멈추는 **GC Spike** 발생 → 프레임 드랍 원인
- **GC 최적화 기법**:
  - `new` 키워드 남용 줄이기 (Heap 할당 최소화)
  - **오브젝트 풀링(Object Pooling)**: 자주 생성/삭제하는 오브젝트(총알, 이펙트 등)를 미리 만들어두고 재사용 → Heap 할당 없음
  - `struct` 사용 (값 타입 → Stack 저장, GC 대상 아님)
- **Stack Overflow**: 재귀 함수가 너무 깊어지면 Stack 영역 초과 → 게임 크래시

## 의문점 / 더 알아볼 것

- C#의 GC는 Heap을 어떤 방식으로 정리하나? (Mark & Sweep)
- Unity의 오브젝트 풀링 구현 방법은? (`UnityEngine.Pool`)
