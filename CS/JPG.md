# JPG (JPEG)

## 핵심 개념

- **Joint Photographic Experts Group**의 약자. 확장자는 `.jpg` 또는 `.jpeg`
- **손실 압축(Lossy Compression)** 방식 — 원본 데이터 일부를 버리고 저장
- 한 번 압축하면 버린 데이터는 복원 불가. 저장할수록 품질 저하 누적

## 손실 압축이 동작하는 방식

사람의 눈이 "잘 못 보는 영역"을 골라서 버리는 원리

```
1. 이미지를 8x8 픽셀 블록으로 분할
2. 각 블록을 주파수 성분으로 변환 (DCT, 이산 코사인 변환)
3. 사람 눈에 덜 민감한 고주파(세밀한 디테일) 성분을 버림
4. 남은 데이터만 저장
```

- 압축률을 높일수록 더 많은 고주파 성분을 버림 → 블록 경계가 뭉개지는 아티팩트 발생
- 이 블록 뭉개짐을 **JPEG 아티팩트**라고 부름

## PNG와 비교

| 항목          | JPG                          | PNG                     |
| ------------- | ---------------------------- | ----------------------- |
| 압축 방식     | 손실                         | 무손실                  |
| 파일 크기     | 작음                         | 큼                      |
| 투명도(Alpha) | 지원 안 함                   | 지원                    |
| 재저장 시     | 품질 저하 누적               | 품질 유지               |
| 적합한 용도   | 사진, 배경                   | UI, 스프라이트, 로고    |
| 색상 표현     | 자연스러운 그라데이션에 강함 | 단색/선명한 경계에 강함 |

- 편집 중인 파일은 PNG로 작업하고 최종 결과물만 JPG로 내보내는 게 맞음
- 원본은 항상 무손실 포맷으로 보관해야 하는 이유

## 게임 개발 관점에서

- **배경 이미지, 스카이박스**: 투명도가 필요 없고 사진에 가까운 이미지는 JPG로 용량 절약 가능
- **투명도 필요한 건 무조건 PNG**: 캐릭터 스프라이트, UI 아이콘, 이펙트 텍스처는 Alpha 채널이 필요하므로 JPG 불가
- **Unity 텍스처 압축과의 관계**: Unity는 임포트 시 JPG/PNG 원본을 내부적으로 GPU용 압축 포맷(DXT, ASTC 등)으로 재변환함. 즉 JPG로 임포트해도 런타임에 쓰이는 포맷은 따로 있음 → 임포트 원본 포맷이 런타임 성능에 직접 영향을 주지 않음
- **텍스처 아틀라스**: 여러 스프라이트를 하나의 텍스처에 모아두는 기법. 투명도가 필요하면 PNG, 배경처럼 투명도 불필요하면 JPG로 아틀라스 저장 가능

## 의문점 / 더 알아볼 것

- Unity의 GPU 압축 포맷(DXT, ASTC, ETC)이 JPG/PNG와 어떻게 다른지
