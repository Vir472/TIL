# PNG (Portable Network Graphics)

## 핵심 개념

- **Portable Network Graphics**의 약자. 확장자는 `.png`
- **무손실 압축(Lossless Compression)** — 압축해도 원본 데이터 완전 보존
- **알파 채널(투명도)** 지원 — 게임 UI/스프라이트의 표준 포맷
- GIF를 대체하기 위해 1996년 만들어진 오픈 포맷 (특허 없음)

## 압축 방식 — Deflate

```
1. 픽셀 데이터에 필터 적용 (인접 픽셀과의 차이값으로 변환)
   → 반복 패턴이 많아져서 압축 효율 올라감
2. Deflate 알고리즘으로 압축 (ZIP과 같은 알고리즘)
3. 결과를 청크(Chunk) 단위로 저장
```

- 무손실이지만 반복 패턴이 많은 이미지일수록 압축률이 높아짐
- 사진처럼 픽셀마다 색이 다른 이미지는 압축률이 낮아 JPG보다 파일이 커짐

## 다른 포맷과 비교

| 항목        | PNG            | JPG        | BMP         | TGA       |
| ----------- | -------------- | ---------- | ----------- | --------- |
| 압축        | 무손실         | 손실       | 없음        | 없음/RLE  |
| 투명도      | 지원           | 미지원     | 미지원      | 지원      |
| 파일 크기   | 중간           | 작음       | 매우 큼     | 큼        |
| 재저장 품질 | 유지           | 저하       | 유지        | 유지      |
| 용도        | UI, 스프라이트 | 사진, 배경 | 중간 작업물 | 3D 텍스처 |

## 색상 깊이(Bit Depth)

```
8bit  — 256색 인덱스 컬러 (구형 PNG, 파일 작음)
24bit — RGB (트루컬러, 투명도 없음)
32bit — RGBA (트루컬러 + 알파 채널)
```

게임에서는 주로 32bit PNG (RGBA) 사용

## PNG 파일 구조 — 청크(Chunk) 기반

```
┌──────────┐
│ 시그니처  │  ← PNG 파일임을 식별하는 8바이트 고정값 (89 50 4E 47 ...)
├──────────┤
│  IHDR    │  ← 이미지 크기, 색상 깊이, 압축 방식 등 헤더
├──────────┤
│  IDAT    │  ← 실제 압축된 픽셀 데이터 (여러 청크로 나뉠 수 있음)
├──────────┤
│  IEND    │  ← 파일 끝을 나타내는 마커
└──────────┘
```

청크 구조라 메타데이터(텍스트, 색상 프로파일 등) 추가가 유연함

## 게임 개발 관점에서

- **UI/스프라이트 표준**: 투명도가 필요한 모든 게임 에셋의 기본 포맷. 캐릭터 스프라이트, 아이콘, 버튼, 이펙트 텍스처 등
- **Unity 임포트**: PNG를 임포트하면 Unity가 내부적으로 GPU 압축 포맷(PC: DXT5, Android: ASTC, iOS: ASTC)으로 변환. 원본 PNG는 에디터용이고 런타임에는 GPU 포맷이 쓰임
- **텍스처 아틀라스**: 여러 스프라이트를 하나의 PNG에 모아서 Draw Call 줄이는 기법. Unity의 Sprite Atlas가 이 방식
- **사진에는 비효율적**: 픽셀마다 색이 다른 사진 이미지는 PNG 압축 효율이 낮아 JPG보다 파일이 훨씬 커짐. 배경 이미지처럼 투명도 불필요한 사진류는 JPG가 나음
- **9-Slice**: UI 버튼처럼 크기가 바뀌어도 모서리가 깨지지 않게 하는 기법. PNG의 투명도와 함께 Unity UI에서 자주 씀

## 의문점 / 더 알아볼 것

- APNG (Animated PNG) — GIF 대체재로 투명도 있는 애니메이션 가능한데 Unity 지원 여부
- GPU 압축 포맷(DXT, ASTC, ETC)이 PNG와 다른 점 — 왜 런타임엔 GPU 포맷을 쓰는지
